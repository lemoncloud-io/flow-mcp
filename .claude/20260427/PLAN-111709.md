## 0. Rethink

**What the user actually wants:**
Claude Desktop (or other MCP clients) 에서 Eureka Flow의 워크플로우를 조회/생성/수정/실행할 수 있는 MCP 서버.

**Why MCP?**
- AI agent가 flow를 직접 조작 → 자연어로 워크플로우 빌드 가능
- n8n-mcp처럼 Claude Desktop에 연결하여 "flow 만들어줘" 식의 대화형 워크플로우 관리

**Scope:**
- Phase 1: Stdio transport (Claude Desktop 연동) + WebSocket 실행 모니터링
- HTTP/SSE transport는 Phase 2 (이번 플랜 범위 밖)

## 2. MCP SDK Version Decision

**Use `@modelcontextprotocol/sdk` v2 (latest)**
- `McpServer` class (not raw `Server`)
- `server.registerTool(name, config, handler)` API
- `inputSchema: z.object({...})` — **must use `import * as z from 'zod/v4'`** (Zod 3.25+ ships v4 as subpath export, required for Standard Schema)
- Error handling: return `{ isError: true, content: [...] }` in handlers

**n8n-mcp는 v1 API 사용 중** (raw `Server` + `setRequestHandler` + `zod` 3.24). 우리는 v2로 가되, 패턴(stdio wrapper, config validation, error handling)은 차용.

> **주의:** `import { z } from 'zod'` (v3 API) 가 아니라 `import * as z from 'zod/v4'` 사용. MCP SDK v2의 `registerTool`이 Standard Schema를 요구하며, Zod v4만 이를 지원.

## 3. Tool Definitions (9 tools)

### Flow Tools

| Tool | Description | API Endpoint |
|------|------------|-------------|
| `flow_list` | List user's flows. Use this first to see available flows. | `GET /flows` |
| `flow_load` | Load flow with full canvas state (nodes, edges, ports). Use to inspect a flow's structure before modifying. | `GET /flows/:id/load` |
| `flow_create` | Create a new flow with nodes and edges. Use block_list first to discover available block types. If edges are provided, handler internally does two-step: create nodes → resolve IDs → save edges. | `POST /flows/0/save` (×2 if edges) |
| `flow_save` | Save flow (full replace of nodes + edges). Always flow_load first to get current state. **Destructive: overwrites all existing nodes/edges.** | `POST /flows/:id/save` |
| `flow_delete` | Delete a flow permanently. | `DELETE /flows/:id` |
| `flow_run` | Execute entire flow. Triggers async, monitors via WebSocket until all nodes complete, returns final status + results. | `POST /flows/:id/run` + WebSocket |

### Node Tools

| Tool | Description | API Endpoint |
|------|------------|-------------|
| `node_run` | Execute a single node (requires flowId). Triggers async, monitors via WebSocket until node completes, returns status + output. | `POST /nodes/:id/run` + WebSocket |
| `node_get_port` | Get port data (execution result) for a node's input or output. | `GET /nodes/:id\::portId@:dir/port` |

### Block Tools

| Tool | Description | API Endpoint |
|------|------------|-------------|
| `block_list` | List all available block definitions with ports and config schema. Call this before creating flows to know what blocks exist and how to configure them. | `GET /blocks/0/list?cores=1` |

> `session_info` 제거 — 디버깅용이며 사용자 가치 없음. 도구 수를 줄여 AI의 도구 선택 정확도 향상.

```ts
class FlowApiClient {
```

**Error Handling Pattern:**
- API 에러 → `{ isError: true, content: [{ type: 'text', text: error.message }] }` 반환
- Network 에러 → 같은 패턴, connection refused 등 구분
- 인증 에러 (401/403) → "API key가 유효하지 않습니다" 메시지
- **[Eng Review 3A] Timeout 에러** → Axios `ECONNABORTED` 감지 시:
  `"Execution timed out (${timeout}ms). The flow may still be running on the server. Use flow_load to check status, or increase FLOW_API_TIMEOUT."` 메시지 반환

## 7. WebSocket Client Design [NEW]

```ts
// src/ws-client.ts
```

### 아키텍처

```
eureka-flows-api (POST /nodes/:id/run)
  → eureka-sockets-api (HTTP broadcast)
    → WebSocket (wss://...?x-api-key=<key>&channels=0000)
      → ws-client.ts (메시지 파싱 + flowId 필터링)
        → 도구 핸들러 (Promise resolve)
```

### 연결 관리

- 도구 호출마다 임시 연결 (connect → listen → disconnect)
- 채널: 항상 `0000` (단일 공유 채널, flowId로 클라이언트 필터링)
- 인증: `x-api-key` 쿼리 파라미터

```ts
const wsUrl = `${FLOW_WS_URL}?x-api-key=${FLOW_API_KEY}&channels=0000`;
```

### 메시지 포맷

외부 envelope:
```json
{ "action": "message", "data": { ... }, "ts": "...", "channel": "0000" }
```

내부 data — 두 가지 래핑 패턴이 존재:

**패턴 A: 서비스 래핑 (broadcast)**
```json
{ "service": "eureka-flows-api", "modelType": "node", "event": "update",
  "data": { "type": "node", "id": "1000637", "flowId": "1000088", "state": "COMPLETED", "ts": ... } }
```

**패턴 B: 직접 이벤트 (일부 케이스)**
```json
{ "type": "node", "id": "1000637", "flowId": "1000088", "state": "COMPLETED", "ts": ... }
```

메시지 파싱은 두 패턴 모두 처리:
```ts
const parseEvent = (envelope: WsEnvelope): FlowEvent | null => {
  const outer = envelope.data;
  // 패턴 A: outer.data.type 존재
  if (outer?.data?.type) return outer.data as FlowEvent;
  // 패턴 B: outer.type 직접 존재
  if (outer?.type) return outer as FlowEvent;
  return null;
};
```

이벤트 타입:
```ts
type FlowEvent =
  | { type: 'node'; id: string; flowId: string; state: string; ts: number; error?: string }
  | { type: 'node/port'; id: string; flowId: string; ts: number }
  | { type: 'flow'; id: string; timestamp: number };
```

### 핵심 함수

```ts
// WebSocket 연결 + 완료 대기 통합 함수
const executeWithWs = (config: WsConfig, params: {
```

### [Eng 1B] Race Condition 방지: Connect-First + Belt-and-Suspenders

```
1. flow_load → 노드 목록 확보 (expectedNodeIds)     [Eng 2A]
2. ws.connect() → open 이벤트 대기
3. POST /flows/:id/run (async) → 실행 트리거
4. 즉시 flow_load → 이미 완료되었는지 확인           [Belt-and-Suspenders]
   └─ 모든 노드가 terminal → resolve (WS 대기 skip)
   └─ 아직 진행 중 → WS 이벤트 대기 계속
5. WS 이벤트 수신 → Map<nodeId, status> 추적
6. expectedNodeIds 전부 terminal → resolve
7. flow_load로 최종 상태 조회
8. ws.close()
```

> **왜 step 4가 필요한가?** 빠르게 실행되는 단일 노드 플로우는 WS 연결 완료 전에 이미 끝날 수 있음.
> step 4의 `flow_load`가 이를 캐치하여 race condition 제거.

### [Eng 2A] 노드 목록 사전 확보

`waitForFlowCompletion`이 "모든 노드 완료"를 판단하려면 전체 노드 수를 알아야 함.
WS 이벤트만으로는 `IDLE`/`SKIPPED` 상태인 노드를 감지할 수 없음.

**해결:** 실행 전 `flow_load`로 노드 ID 목록을 가져와 `expectedNodeIds`로 전달.

```ts
// flow_run handler 내부
const flow = await client.loadFlow(flowId);
const expectedNodeIds = (flow.nodes ?? [])
  .filter(n => !n.disabled)
  .map(n => n.id!)
  .filter(Boolean);
```

### [Eng 3A] node_run에 flowId 필수 파라미터 추가

기존: `nodeId`만 입력
변경: `nodeId` + `flowId` 필수 입력

이유: WS 필터링에 `flowId` 필요. AI는 이미 `flow_load`로 flowId를 알고 있음.
추가 API 호출 없이 WS 구독 가능.

```ts
input: z.object({
  nodeId: z.string().describe('Node ID to execute'),
  flowId: z.string().describe('Flow ID that contains this node. Get from flow_load.'),
  propagate: z.optional(z.boolean()).describe('Propagate to downstream nodes'),
  config: z.optional(z.record(z.string(), z.string())),
  timeout: z.optional(z.number()).describe('Max wait in ms (default: 30000)'),
})
```

### 동작 흐름 (flow_run) — 최종

```
1. flow_load(flowId) → expectedNodeIds 확보
2. ws.connect() → open 대기
3. POST /flows/:id/run (async)
4. flow_load(flowId) → 이미 완료? → 즉시 반환
5. WS 이벤트 대기 (flowId 필터, node type만)
6. Map<nodeId, status> 추적, expectedNodeIds 전부 terminal → resolve
7. flow_load(flowId) → 최종 상태 조회
8. ws.close()
9. 반환: { flowId, status, nodes, duration }
```

### 동작 흐름 (node_run) — 최종

```
1. ws.connect() → open 대기
2. POST /nodes/:id/run (async, propagate 옵션)
3. WS 이벤트 대기 (flowId 필터, nodeId 매칭)
4. COMPLETED/ERROR → resolve
5. node_get_port로 출력 수집
6. ws.close()
7. 반환: { nodeId, status, error?, outputPorts, duration }
```

### Timeout 처리

| 도구 | 기본 | 동작 |
|------|------|------|
| flow_run | 60초 | 현재 노드 상태 스냅샷 + "N/M 노드 완료, 나머지 진행 중" |
| node_run | 30초 | 현재 상태 반환 + "아직 실행 중. flow_load로 확인하세요" |

### Fallback (FLOW_WS_URL 없을 때)

```
FLOW_WS_URL 설정됨 → WebSocket 모드 (위 흐름)
FLOW_WS_URL 없음   → Sync fallback (기존 async=0 동작)
```

## 8. Stdio Wrapper (Critical)

n8n-mcp의 3-layer protection 패턴 적용:

```ts
// src/stdio.ts
// Layer 1: env flag
process.env.MCP_MODE = 'stdio';

// Layer 2: console.* suppression
console.log = () => {};
console.error = () => {};
// ... all methods

// Layer 3: stdout.write intercept (JSON-RPC only)
const originalWrite = process.stdout.write.bind(process.stdout);
process.stdout.write = (chunk, ...rest) => {
  const str = typeof chunk === 'string' ? chunk : chunk.toString();
  if (str.trimStart().startsWith('{') && str.includes('"jsonrpc"')) {
    return originalWrite(chunk, ...rest);
  }
  return process.stderr.write(chunk, ...rest);
};

// Import AFTER suppression
import('./server.js').then(m => m.createServer().run());
```
